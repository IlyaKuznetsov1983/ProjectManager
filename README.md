                                            Часть Первая
                            Идентификация проблем
 Отсутствие приоритизации и scope creep
СЕО постоянно добавляет новые фичи (монетизация, иконка, питание) без четких приоритетов, что размывает фокус команды.
 Блокирующие зависимости между командами
Frontend и backend разработчики не синхронизированы (нет API для авторизации), что останавливает прогресс.
  Неопределенность требований и критериев готовности
Дизайнер создает макеты без контента, тестировщики работают без четких тест-кейсов.
 Отсутствие прозрачности процесса и плана релиза
Нет четкого roadmap, команда не понимает сроков и целей, маркетинг планирует активности вслепую.
 Технические долги и качество кода
Падающая база данных и рефакторинг вместо разработки новых фич снижают скорость и мотивацию.
                                           Часть Вторая
                     Выбор методологии и первые шаги
                                    Рекомендуемая методология
Scrum/Kanban гибрид - 2-недельные спринты с ежедневными стендапами.
Scrum обеспечит структуру и регулярное планирование, а Kanban-доска визуализирует блокеры. Это решит проблемы приоритизации и прозрачности.
                                   Повестка первой встречи
Выявление всех текущих блокеров - что прямо сейчас мешает двигаться?
Определение MVP - что минимальное нужно для запуска?
Создание единого backlog с приоритизацией от СЕО
Договоренность о процессе - как будем работать дальше?
                           Бэклог первых задач
Разработать и предоставить API для авторизации (бэкенд)
Исправить критические базы данных для стабильной работы тестового стенда
Определить и утвердить MVP с СЕО (список обязательных фич для первого релиза)
Создать детальные требования для раздела "Тренировки" (контент + дизайн)
Настроить CI/CD пайплайн для стабильных билдов
Этот план остановит "пожары", создаст прозрачность и вернет команде уверенность в достижимых целях.
